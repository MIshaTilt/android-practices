# Отчет по практической работе №6
**«Хранение данных в OS Android»**

### 1. Цель работы
Ознакомиться с основными способами локального хранения данных в операционной системе Android. Получить практические навыки работы с файлом настроек `SharedPreferences` (в том числе с его зашифрованной версией `EncryptedSharedPreferences`), потоковой записи и чтения файлов во внутреннем и внешнем хранилищах устройства. Изучить архитектурный компонент `Room` для структурированной работы с локальной реляционной базой данных `SQLite` и реализовать базовые CRUD-операции.

### 2. Программная реализация

В ходе выполнения практической работы проект был разбит на несколько независимых модулей, каждый из которых демонстрирует отдельный механизм хранения данных.

#### 2.1. Работа с SharedPreferences (модуль `app`)
**Задача:** Сохранить пользовательские данные (номер группы, номер по списку, любимый фильм) с автоматическим восстановлением при перезапуске приложения.
**Объяснение кода:**
Для хранения небольших структур данных в формате «ключ-значение» использовался класс `SharedPreferences`.
*   Инициализация происходила с помощью метода `getSharedPreferences("mirea_settings", Context.MODE_PRIVATE)`, где создается или открывается приватный XML-файл настроек.
*   Для записи данных вызывался метод `edit()`, возвращающий объект `SharedPreferences.Editor`. В него передавались значения из текстовых полей `EditText` с помощью `putString()`. Сохранение выполнялось асинхронным методом `apply()`, что предотвращает блокировку главного потока (UI-потока).
*   При запуске активности метод `loadData()` использовал `getString()` для извлечения сохраненных данных и установки их обратно в элементы интерфейса. В качестве второго аргумента передавалась пустая строка (значение по умолчанию на случай отсутствия данных).

#### 2.2. Работа с EncryptedSharedPreferences (модуль `SecureSharedPreferences`)
**Задача:** Записать и прочитать данные из зашифрованного файла настроек, отобразить на экране фотографию и имя любимого поэта.
**Объяснение кода:**
Для защиты конфиденциальных данных от прямого чтения (даже при наличии root-прав) была подключена библиотека `androidx.security:security-crypto`.
*   Сначала был сгенерирован мастер-ключ с использованием алгоритма AES256-GCM: `MasterKeys.getOrCreate(MasterKeys.AES256_GCM_SPEC)`.
*   Затем был создан экземпляр `EncryptedSharedPreferences`. При его создании были указаны схемы шифрования: `AES256_SIV` для ключей и `AES256_GCM` для значений.
*   Операции чтения и записи в коде (`putString` и `getString`) выглядят так же, как и в обычных `SharedPreferences`, однако «под капотом» библиотека автоматически выполняет криптографические преобразования. При извлечении файла из эмулятора (через Device File Explorer) было наглядно продемонстрировано, что данные хранятся в виде нечитаемого шифротекста.

#### 2.3. Внутреннее хранилище (модуль `InternalFileStorage`)
**Задача:** Сохранить введенную текстовую информацию (памятная историческая дата) в текстовый файл, расположенный в приватной директории приложения.
**Объяснение кода:**
*   Для создания файла и открытия потока записи применялся метод `openFileOutput("history_date.txt", Context.MODE_PRIVATE)`. Флаг `MODE_PRIVATE` гарантирует, что доступ к файлу будет только у текущего приложения.
*   Текст из `EditText` преобразовывался в массив байт методом `getBytes()` и записывался через объект `FileOutputStream`. В блоке `finally` (или после записи) поток `close()` закрывался для освобождения системных ресурсов.
*   Файл успешно сохранился в защищенной системной директории `/data/data/<package_name>/files/`.

#### 2.4. Внешнее хранилище (модуль `Notebook`)
**Задача:** Реализовать приложение «Блокнот» с возможностью сохранения цитат в общедоступную файловую систему устройства (внешнее хранилище) и их последующей загрузки.
**Объяснение кода:**
*   В `AndroidManifest.xml` были добавлены необходимые разрешения: `WRITE_EXTERNAL_STORAGE` и `READ_EXTERNAL_STORAGE`.
*   Для работы с файлами во внешней памяти использовались стандартные классы Java I/O (`FileOutputStream`, `FileInputStream`, `OutputStreamWriter`, `BufferedReader`).
*   Код был обернут в конструкцию `try-catch` для обработки возможных исключений (например, `IOException` при нехватке места или отсутствии разрешений).

#### 2.5. База данных SQLite с использованием Room (модуль `EmployeeDB`)
**Задача:** Создать локальную реляционную базу данных для хранения информации о вымышленных супергероях, используя ORM-компонент Room.
**Объяснение кода:**
Архитектура Room потребовала разделения логики на несколько ключевых компонентов:
1.  **Entity (`Superhero.java`)**: Класс-модель (POJO), помеченный аннотацией `@Entity`. Поля класса стали столбцами таблицы. Поле `id` помечено как `@PrimaryKey(autoGenerate = true)` для автоматического инкремента.
2.  **DAO (`SuperheroDao.java`)**: Интерфейс взаимодействия с БД (`Data Access Object`). Содержит аннотированные методы `@Insert`, `@Update`, `@Delete` и `@Query("SELECT * FROM superhero")`. Аннотации позволили избежать написания громоздкого SQL-кода вручную, Room сгенерировал реализацию на этапе компиляции.
3.  **Database (`AppDatabase.java`)**: Абстрактный класс, наследующий `RoomDatabase`. В аннотации `@Database` указан список сущностей (`entities = {Superhero.class}`) и версия схемы БД (`version = 1`).
4.  **Application (`App.java`)**: Класс, инициализирующий базу данных в методе `onCreate()` при запуске приложения (паттерн Singleton). Зарегистрирован в `AndroidManifest.xml`.
*   В `MainActivity` был реализован алгоритм добавления нового героя ("Бэтмен") в БД (`insert`), получения его по ID (`getById`), изменения его свойств (смена суперсилы) и обновления записи в БД (`update`). Успешное выполнение операций фиксировалось с помощью инструмента логирования `Log.d` в Logcat.

### 3. Заключение
В ходе выполнения практической работы были успешно изучены и применены на практике все ключевые механизмы хранения данных в ОС Android. Были получены навыки сохранения простых данных и настроек (`SharedPreferences`), обеспечения их криптографической безопасности с помощью Jetpack Security (`EncryptedSharedPreferences`), а также работы с файловой системой устройства.

Особое внимание было уделено архитектурному компоненту `Room`, с помощью которого была спроектирована и реализована локальная база данных на базе SQLite. Использование DAO, Entity и строгого паттерна инициализации позволило безопасно и эффективно реализовать базовые операции (CRUD) создания, чтения и обновления данных. Все поставленные задачи выполнены в полном объеме.
